# Admin UI

The dashboard is intentionally vanilla — three HTML pages, three JS files,
one CSS file, no build step. Everything lives in
[`admin_ui/`](../admin_ui/) and is served directly by FastAPI route
handlers in [`admin_api.py`](../admin_api.py).

## File layout

```
admin_ui/
├── dashboard.css      # Single stylesheet for every page
├── home.html          # Landing page after Okta login
├── home.js            # Loads the user's profile + recent activity
├── index.html         # "Dashboard" — model / agent / message editor
├── dashboard.js       # Wires CRUD calls against /admin/models, /admin/agents, /admin/messages
├── responses.html     # "Responses" — query log explorer
├── responses.js       # Loads /admin/query-logs, exports CSV, applies blacklist/whitelist
└── login.html         # Standalone preview shown on /admin/ when not authenticated
```

## Routing

The dashboard pages map to these admin routes:

| Page                     | Route                       | Source             |
|--------------------------|-----------------------------|--------------------|
| Landing / "Home"         | `GET /admin/`               | `home.html`        |
| Configuration dashboard  | `GET /admin/dashboard`      | `index.html`       |
| Query log explorer       | `GET /admin/responses`      | `responses.html`   |

Static assets are mounted under `/admin/static/*` and served by explicit
`@admin_router.get` handlers (not `StaticFiles`) so that the route prefix
and Okta auth interact cleanly.

## Why these paths use *relative* asset hrefs

Each HTML file references CSS/JS with relative paths like
`href="static/dashboard.css"` instead of absolute paths. That keeps the
same markup working in three contexts:

1. Local dev — served at `http://localhost:7777/admin/`.
2. Platform.sh — served at `https://www.colby.edu/chatbot-api/admin/`.
3. Behind any other reverse proxy that may rewrite the prefix.

Combined with `app.root_path = "/chatbot-api"` (set in `rag_app.py`), URLs
generated by `request.url_for` also pick up the prefix automatically.

## API consumption

All three JS files are simple `fetch` consumers of the admin API:

- `home.js` → reads recent activity + user info.
- `dashboard.js` → reads/writes `llm_models`, `llm_agents`,
  `agent_instructions`, `app_messages`. Validates a typed-in model id via
  `POST /admin/validate-model`.
- `responses.js` → paginates `GET /admin/query-logs`, calls
  `/admin/query-logs/{id}/blacklist` and `/whitelist` for promote/demote,
  triggers CSV download via `GET /admin/query-logs/export/csv`.

There is no SPA framework, bundler, or state manager — the dashboard
trades local complexity for matching the operational simplicity of the
rest of the API container.
